二.使用

开始

  1. Vuex 应用的核心就是 store（仓库）,里面储存着数据和方法.
  2. 当组件通过store拿到数据并进行更改,store中数据也会实时更改.因此不建议组件中更改store中的数据,建议在mutation中改.这样能方便跟踪每一个状态的变化.若在组件中直接改了就无法跟踪状态.(就比如多个组件同时要用仓库的数据,但是某个组件错误操作了数据对数据进行更改 产生了数据紊乱 没法跟踪状态 你怎么知道是哪个组件错误操作了呢 多个组件且代码量多的情况下一个个找,不容易快速定位,也不利于代码维护)

核心概念

1.vuex的核心概念是:

  • State 译状态 相当于data数据
  • Getter 译获得者 相当于computed计算属性
  • Mutation 译变化 相当于method方法 写同步方法 主要是操作data中的数据
  • Action 译行动 写异步方法
  • Module 译模块 大型项目时所有数据都写在一个文件里代码量臃肿 不利于维护 所以可将其分成多个模块

2.3 Mutation

Mutation类似methods,主要写同步方法,更改 Vuex 的 store 中的状态的唯一方法是提交 mutation,例若要更改state数据 就在这里面操作 可以监听状态变化,如果在组件中操作就监听不了.
(千万不要在组件中直接修改Vuex的数据,这不符合Vuex的设计理念,如果要改 就通过调用mutation改,前面也有提到组件中直接操作数据并更改 是无法跟踪状态的 如果某个组件错误操作数据 让数据紊乱 在不能跟踪状态变化的前提下 你如何知道是哪个组件错误操作的呢,多个组件且代码量多的情况下一个个找,不容易快速定位,也不利于代码维护 )
它接受 state 作为第一个参数 这里的state相当于store.state

因为当mutation触发的时候就被记录了,但是此时回调函数还没调用,所以显示的是改变前的结果.devtools 不知道什么时候回调函数实际上被调用——实质上任何在回调函数中进行的状态的改变都是不可追踪的。

2.3.4在组件中提交 Mutation(推荐用mapMutations)

你可以在组件中使用 this.$store.commit(‘xxx’) 提交 mutation，或者使用 mapMutations 辅助函数将组件中的 methods 映射为 store.commit 调用（需要在根节点注入 store）

2.3 Action

Action 类似于 mutation，不同在于：
Action 提交的是 mutation，而不是直接变更状态。
Action 可以包含任意异步操作。

Action接收第一个参数context译上下文,是与 store 实例具有相同方法和属性的 context 对象,因此你可以调用 context.commit 提交一个 mutation，或者通过 context.state 和 context.getters 来获取 state 和 getters。
当需要多次用到commit时通常用赋值解构

module

由于使用单一状态树，应用的所有状态会集中到一个比较大的对象。当应用变得非常复杂时，store 对象就有可能变得相当臃肿。
Vuex 允许我们将 store 分割成模块（module）。每个模块拥有自己的 state、mutation、action、getter、甚至是嵌套子模块——从上至下进行同样方式的分割：

注意,
\1. 在模块中写的Getter默认是在全局上的 所以不能通过模块的方式this.$ route.state.a访问 只能通过在全局的方式访问 例this.$route.getters.a
\2. 模块内的参数例state,getters,context都是针对模块局部的
